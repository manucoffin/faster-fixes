---
title: Self-hosting
description: Deploy FasterFixes on your own infrastructure. Host the dashboard, the widget API, and the MCP server without relying on the cloud version.
---

FasterFixes is AGPL-3.0. You can deploy the full stack — dashboard, widget API, and background jobs — on your own infrastructure. This guide covers the minimal setup for a production deployment.

The widget packages (`@fasterfixes/core`, `@fasterfixes/react`) are MIT-licensed and can be embedded in any client project regardless of where you host the backend.

## Requirements

You will need:

- A **Node.js host** — Vercel is recommended (the repo is a Next.js app), but anything that runs Next.js will work (Railway, Fly, self-managed).
- A **PostgreSQL database** — Neon, Supabase, Railway, or self-managed.
- An **Inngest** account for background jobs (screenshot processing, GitHub sync, etc.). The free tier is enough to get started.
- An **S3-compatible object store** for screenshots — Cloudflare R2 is the recommended default; AWS S3 works identically.
- A **Resend** or **Plunk** account for transactional email — required to send sign-up confirmations and password resets.
- (Optional) A **GitHub App** if you want the GitHub integration.

Stripe is not needed for self-hosted installs. The billing code only runs on the hosted cloud version.

## 1. Clone and deploy

Fork or clone [the repository](https://github.com/manucoffin/faster-fixes), then deploy the `apps/web` workspace to your host.

```bash
git clone https://github.com/manucoffin/faster-fixes.git
cd faster-fixes
pnpm install
```

On Vercel, import the repo and set the root directory to `apps/web`. Vercel auto-detects Next.js and Turborepo.

## 2. Configure environment variables

Copy `apps/web/.env.example` and fill in the values. The minimum set for a self-hosted deployment:

```bash
# Core
DOMAIN_NAME="your-domain.com"
BASE_URL="https://your-domain.com"
DATABASE_URL="postgresql://user:password@host:5432/db?schema=public"

# Auth
BETTER_AUTH_SECRET="<generate with: openssl rand -base64 32>"
BETTER_AUTH_URL="https://your-domain.com"

# Storage (Cloudflare R2)
STORAGE_REGION="auto"
STORAGE_BUCKET_NAME="your-bucket"
NEXT_PUBLIC_STORAGE_BASE_URL="https://<account-id>.r2.cloudflarestorage.com/your-bucket"
R2_ACCOUNT_ID="..."
R2_ACCESS_KEY_ID="..."
R2_SECRET_ACCESS_KEY="..."

# Inngest (required in production)
INNGEST_EVENT_KEY="..."
INNGEST_SIGNING_KEY="..."

# Transactional email — required for sign-up confirmations and password resets.
# Use either Resend or Plunk.
RESEND_API_KEY="re_..."
# PLUNK_SECRET_KEY="sk_..."

# Widget API (points clients' widgets at your instance)
NEXT_PUBLIC_FF_API_ORIGIN="https://your-domain.com"

# Self-hosted flag (keep false to skip marketing pages)
NEXT_PUBLIC_IS_CLOUD="false"
```

GitHub App variables are optional. Skip them unless you need the GitHub integration. Leave the Stripe variables unset — billing is cloud-only and self-hosted installs don't run it.

If you prefer AWS S3 over Cloudflare R2, use `AWS_ACCESS_KEY_ID`, `AWS_BUCKET_NAME`, and `AWS_SECRET_ACCESS_KEY` instead of the R2 variables.

## 3. Provision external services

### PostgreSQL

Any Postgres 14+ works. Create a database and put the connection string in `DATABASE_URL`.

### Cloudflare R2

Create a bucket and an API token scoped to that bucket. Copy the account ID, access key, and secret into the R2 variables above. Make the bucket public (or front it with a Worker) so screenshots are readable by reviewers.

### Inngest

Create an app at [inngest.com](https://www.inngest.com). Copy the event key and signing key. After your first deployment, connect Inngest to your instance at `https://your-domain.com/api/inngest`.

## 4. Run migrations

From the repo root:

```bash
pnpm --filter @workspace/db migrate:dev
```

This applies all Prisma migrations and generates the client. For subsequent deploys, your host's build step should run `prisma generate`; Prisma migrations are committed to the repo and applied via `migrate:dev` in development and `migrate:deploy` in production (run this as part of your deploy pipeline).

## 5. Verify

After the first deploy:

1. Visit `https://your-domain.com/login` and create the first account (becomes the admin).
2. Create a project, copy the widget API key, and test the widget in any page.
3. Check Inngest's dashboard to confirm events are being received.

Your instance is now ready. Point your clients' widgets at `NEXT_PUBLIC_FF_API_ORIGIN` and connect your AI coding agent via the [MCP server](/docs/mcp/setup).
