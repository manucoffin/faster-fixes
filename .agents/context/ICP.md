# Ideal Customer Profile (ICP)

## Overview

This project targets **small developer-led web agencies** that build SaaS or web applications for **non-technical clients**.

The product helps these teams handle **messy client feedback** during the review and maintenance phases of a project by turning it into **clear, actionable developer tasks**.

The core value is:

> **messy client feedback → structured developer tasks**

---

# Target Customer

## Company Profile

Small web agencies with:

- **2–10 employees**
- **3–5 active client projects at the same time**

Typical team composition:

- 1 lead developer or CTO
- 2–4 developers
- 1 designer
- sometimes a project manager or founder involved in delivery

These agencies typically build:

- SaaS products
- internal dashboards
- marketplaces
- web applications

for **external, non-technical clients**.

---

# Buyer Persona

The typical buyer is a **technical delivery lead**, such as:

- Lead Developer
- Agency CTO
- Technical Project Manager
- Senior Full-Stack Developer responsible for delivery

Characteristics:

- very pragmatic
- prefers simple tools
- dislikes heavy project-management systems
- values **developer-first tooling**

They tend to like tools such as:

- Vercel
- Resend
- Supabase
- Linear
- Stripe

In other words: **modern developer tooling with strong DX (developer experience)**.

# Client Profile

Their clients are typically **non-technical stakeholders**, such as:

- startup founders
- product managers
- marketing teams
- business executives

These clients:

- are not comfortable with developer tools
- communicate visually
- often give feedback in messy ways

Typical feedback channels today:

- WhatsApp
- Slack
- email
- screenshots
- Google Docs
- Loom videos

---

# When the Product Is Used

The product becomes useful **as soon as the client can access the application**, typically during:

- preview environments
- staging
- production

However, the most critical moment is the **review / acceptance phase**.

Typical duration:

**2–4 weeks**

During this period:

- the client discovers the product
- many UI/UX comments appear
- multiple issues are reported
- feedback becomes chaotic

---

# Type of Feedback

Feedback is usually **not only bug reports**.

Common feedback includes:

- UI tweaks
- wording changes
- layout spacing
- colors
- UX behavior
- small feature adjustments

Because of this, traditional bug trackers are often **not well suited**.

The product acts more like a **feedback layer on top of the website**.

---

# Core Pain Point

The main pain point is **not bugs**.

It is the **time and cognitive load required to interpret client feedback**.

Examples of messy feedback:

- long WhatsApp messages with dozens of comments
- many short messages sent separately
- phone photos of screens instead of screenshots
- no link to the page
- no reproduction steps
- unclear priorities

Developers must then:

1. understand the feedback
2. find the correct page
3. reinterpret the request
4. create a technical task
5. implement the change

This process wastes time and creates mental overhead.

Most agencies currently rely on ad-hoc communication tools:

- WhatsApp
- Slack
- email

- Loom
- manual screenshots

These solutions are:

- chaotic
- unstructured

---

# Why Existing Tools Are Often Not Used

Many small agencies avoid tools like BugHerd or Marker because they are perceived as:

1. **Too expensive**

Typical pricing:
$80–$120/month

2. **Too complex**

Many features oriented toward QA teams or enterprises.

3. **Heavy onboarding**

Accounts, permissions, setup.

4. **Not developer-first**

More oriented toward project managers than developers.

---

# Job To Be Done

The product does **not simply annotate websites**.

Its real job is:

> **Transform messy client feedback into clear developer tasks.**

---

# Ideal Workflow

Current workflow:

Client  
→ sends messages on WhatsApp / Slack  
→ developer interprets feedback  
→ creates a ticket  
→ implements fix

Target workflow:

Client  
→ leaves contextual feedback on the website  
→ system captures context automatically  
→ feedback becomes a structured task  
→ developer implements the fix

Long-term vision:

Client feedback  
→ structured task  
→ AI coding agent  
→ code fix

---

# Value Proposition

Core positioning:

**Turn messy client feedback into clean developer tasks.**

Alternative formulations:

- **Client feedback that becomes dev-ready tasks**
- **From client comment to developer action**

AI features are secondary and come **after solving the workflow problem**.
